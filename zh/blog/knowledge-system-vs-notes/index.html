    开发者需要的不是笔记堆，而是知识系统

    多数开发者已经在记笔记。真正的短板不在「能不能记下来」，而在「需要时能不能找回来」——三个月后再排查同一个 bug、带新人接手，或把两个项目的想法串起来时，差距就会显现。

    笔记存信息，系统存关联

    一条笔记可能写着「Redis 缓存 TTL 是 300 秒」。知识系统还会把它链到 [[auth-service]]、[[incident-2025-11]]，以及你的缓存失效运维手册。当其中任一上下文发生变化，你顺着链接就能追过去，而不必从零开始搜索。

    一套够用的知识系统，最少需要什么

    • 纯文本文件 — 磁盘上的 Markdown，可用 Git 管理版本，用任意工具同步。
    • 双链 — [[笔记标题]] 在不依赖僵化文件夹的前提下，把相关想法连起来。
    • 反向链接 — 查看笔记库里还有哪些地方引用了同一个概念。
    • 知识图谱 — 发现主题聚类、孤立笔记，以及概念之间缺失的桥梁。
    不必第一天就搭完美分类体系。写完下一条笔记时，给它链到一条相关笔记——这就已经是一个系统了。

    为什么开发者会先尝到苦果

    代码库本身就是一张网：模块互相引用，服务互相调用，文档指向工单。开发者习惯用「引用」来思考——但传统线性笔记应用会把这种结构藏起来。Lunote 这类工具把同样的链接模型带到个人文档里，同时又不把文件锁进专有数据库。

    不必推倒重来

    直接打开你已经在用的文件夹——Obsidian 笔记库、Typora 项目，或某个 docs/ 目录。在原本会粘贴 URL、或复制整段文字的地方，改成写一条双链。让反向链接告诉你：哪些笔记该有更清晰的标题，或开头加一段简短摘要。

    用你现有的 .md 文件夹试试

    Lunote 可直接打开本地 Markdown 笔记库，内建双链、反向链接与知识图谱——无需导入。
