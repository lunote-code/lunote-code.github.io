    Markdown 文件够了吗？何时需要知识系统

    Markdown 能赢，是因为它永远可读：方便 diff、方便搜索、也不绑死某个工具。下一层不必是专有数据库——而是在同一批文件上，加一层轻量的链接关系。

    文件夹撑不起知识结构

    把笔记放进 backend/redis/，等于强行指定一个主分类。但真实主题往往跨层：缓存会牵涉 auth、计费和运维。双链能表达多对多关系，而不必复制内容。

    什么应该留在文件里

    • 正文、代码块与图片——标准 Markdown。
    • 可选的 YAML Frontmatter，用来存标签、状态或日期。
    • [[双链]]，以及 ![[相关笔记]] 这类嵌入（在支持的应用里）。

    应用在读文件时额外提供什么

    • 反向链接 — 还有哪些笔记指向这里？
    • 知识图谱 — 这条笔记属于哪个主题聚类？
    • 跨标题、正文与标签的全文搜索。
    • 重命名联动 — 笔记改名或移动时，自动更新链接目标。

    怎么选 Markdown 原生应用

    警惕被工具绑架：如果明天不能用别的编辑器打开这个文件夹，你其实是在「租用」笔记库。Lunote 和 Typora、Obsidian 一样，把目录当作笔记库——但更强调顺手的写作体验，以及内建的双链与导出，不必先堆一圈插件才有图谱或 PDF、Word 导出。

    最好的知识系统，是那种就算应用明天消失，你也依然想保留的文件。

    Markdown + 双链，一个应用就够

    在 Lunote 里打开你的笔记库——Typora 式写作，Obsidian 式双链。
